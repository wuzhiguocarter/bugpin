# Changelog 1.0.24

修复 admin 后台手动建/激活的账号无法被指派的问题（lula 2026-06-01）。

## 🐛 Fixes

### 放宽 findAssignable / isAssignable / getAssignableById 的邀请门

**问题**：账号 `imluli.me@gmail.com`（lula 自己的 admin 账号）发现指派下拉里看不到自己。根因——

老逻辑 `findAssignable` 的 WHERE 条件：
```sql
WHERE is_active = 1
  AND (invitation_sent_at IS NULL OR invitation_accepted_at IS NOT NULL)
```

`isAssignable(user)` 同样：
```ts
user.isActive && (!user.invitationSentAt || !!user.invitationAcceptedAt)
```

`getAssignableById` 在 `invitation_sent_at && !accepted` 时直接拒绝并返回 `INVITATION_PENDING`。

但对于「admin 后台手动建账号 / 邀请发出但 SMTP 失败 / admin 手动激活已存在邀请用户」的场景，会出现 `is_active=1` 但 `invitation_sent_at` 已设值的状态，于是被三条规则同时排除。这就是 lula 在前几版（v1.0.21~23）一直看不到自己 + 报 `INVALID_ASSIGNEE` 的根因。

**改动**：

| 位置 | 改前 | 改后 |
|---|---|---|
| `users.repo.ts findAssignable` SQL | `is_active=1 AND (sent IS NULL OR accepted IS NOT NULL)` | `is_active = 1` |
| `users.service.ts isAssignable` | `isActive && (!sent || !!accepted)` | `isActive` |
| `users.service.ts getAssignableById` | 拒绝 `INVITATION_PENDING` | 移除该检查 |
| `users.service.ts update willBeAssignable` | 同前 | 改为 `updatedIsActive` |

**为什么安全**：
- 真正未激活的邀请用户 `is_active = 0`，仍然被排除
- admin 把账号置为 `is_active = 1` 就是信号「这个人能接活」，邀请门反而是双重门
